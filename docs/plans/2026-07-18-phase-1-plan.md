## Global constraints

- Package name `electron-sparkle-updater`, ESM only (`"type": "module"`), TypeScript compiled with `tsc` to `dist/`. Node >= 20 (`engines`). Dev tooling: pnpm, vitest.
- `electron` is a peer dependency and must never be imported at module top level in `src/` — only lazily inside convenience helpers, so the core stays loadable in plain Node for tests.
- The npm `install` script MUST be the no-op `node -e \"\"` with an adjacent comment in README explaining why (npm auto-runs `node-gyp rebuild` for any package containing a `binding.gyp`, against the host Node ABI — the wrong target; the real build is the explicit `rebuild` command against the Electron ABI).
- Sparkle version pinned: 2.9.4, tarball sha256 `ce89daf967db1e1893ed3ebd67575ed82d3902563e3191ca92aaec9164fbdef9`.
- Zero code comments except where the source files being extracted already carry a load-bearing comment (workaround/invariant); those move with the code. No JSDoc on internal functions.
- All docs and identifiers in English.
- Every task: TDD where there is logic to test (loader path resolution, CLI arg parsing, builder fragment generation); plain verification (script runs, tsc passes) where there is not. Commit per task on branch `phase-1`.

## Task 3: Runtime loader (`src/index.ts`)

Generalize `/Users/innei/git/innei-repo/trade-skills/apps/desktop/src/updater/sparkle.ts`:

- Types: `SparkleInitOptions { appcastUrl: string; publicEdKey: string }`, `SparkleBridge { init(options): boolean; checkForUpdates(): void; installUpdateNow(): void; setAutomaticChecks(enabled: boolean): void }`.
- `resolveSparkleAddonPath(deps: { isPackaged: boolean; resourcesPath: string; packageRoot: string })` — the addon now lives inside this package: dev → `<packageRoot>/native/build/Release/sparkle_bridge.node`; packaged → `<resourcesPath>/app.asar.unpacked/node_modules/electron-sparkle-updater/native/build/Release/sparkle_bridge.node`.
- `loadSparkleBridge(deps: { isPackaged: boolean; resourcesPath: string; log?: (m: string) => void; addonPath?: string })`:
  - returns `null` immediately when `process.platform !== "darwin"` (before any path work)
  - `addonPath` override wins when provided (escape hatch for exotic layouts)
  - default packageRoot derived from this module's own `import.meta.url` (dist/index.js → package root is one level up)
  - loads via `createRequire`, validates all four methods are functions, returns `null` on any failure (never throws), logging through `log`
- `loadSparkleBridgeForApp(log?)` convenience: lazily `await import("electron")`, fills `isPackaged`/`resourcesPath` from `app`/`process.resourcesPath`. This is the only place electron is imported.
- Tests (vitest, TDD): platform guard (non-darwin returns null — stub `process.platform`), addonPath override, path resolution for dev vs packaged layouts, missing-addon returns null and logs, addon-with-missing-methods returns null. Use a fixture directory with a fake `.node` (CommonJS file exporting the four functions) to test the happy path without the real native build.
- Verify: `pnpm test`, `pnpm typecheck`, `pnpm build`. Commit.

## Task 4: `rebuild` CLI command

Implement `bin/electron-sparkle-updater.js` as a thin ESM entry that imports `dist/cli.js`; implement `src/cli.ts`:

- Command: `electron-sparkle-updater rebuild [--electron-version <v>] [--arch arm64|x64|universal] [--force-fetch]`
  - `--electron-version` omitted → resolve from the consumer project: `createRequire(process.cwd() + "/package.json")` then `require("electron/package.json").version`; error with a clear message if electron is not resolvable.
  - `--arch` omitted → `process.arch`.
  - `universal` → run node-gyp with `--arch=arm64` then `--arch=x64` into separate build dirs and `lipo -create` the two `.node` files into `native/build/Release/sparkle_bridge.node`. If that proves awkward inside node-gyp's fixed build dir, building twice and copying the intermediate before the second run is acceptable — document whichever shape ships in the README section (Task 6 covers prose; leave a one-line pointer).
  - Steps: refuse on non-darwin; run `native/scripts/fetch-sparkle.sh` (skip logic lives in the script; `--force-fetch` removes the stamp first); run `node-gyp rebuild --target=<version> --arch=<arch> --dist-url=https://electronjs.org/headers` with `cwd` = this package's `native/` dir (resolve via `import.meta.url`, works when the package sits in the consumer's `node_modules`).
  - Spawn with `stdio: inherit`; non-zero child exit → non-zero process exit.
- Arg parsing with `node:util` `parseArgs` — no new dependencies.
- Tests (TDD): arg parsing (defaults, explicit values, unknown command → usage + exit 1, bad arch value rejected), electron-version resolution failure message. Spawn/exec goes through an injected runner so tests never launch real builds.
- Verify: `pnpm test` + run the real thing once: `node bin/electron-sparkle-updater.js rebuild --electron-version 43.1.0 --arch arm64` from the repo root succeeds end to end (vendor already present from Task 2 makes this fast). Commit.
